# CLI 到 Provider 演进蓝图

从"本地工具"向"平台服务"的跨越，分四个阶段实现平稳演化。

## 目标架构

采用混合模式，服务端为事实源，Git 仓库保留可见性：

```
┌─────────────────────────────────────────────────────────────┐
│                    Provider 服务端                          │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────────────┐ │
│  │   Contract  │  │   Catalog   │  │   项目-仓库映射      │ │
│  │   数据库    │  │   历史库    │  │   (Project-Repo)    │ │
│  │  (事实源)   │  │  (快照历史)  │  │                     │ │
│  └──────┬──────┘  └─────────────┘  └─────────────────────┘ │
│         │                                                   │
│         │  同步                                              │
│         v                                                   │
│  ┌─────────────┐                                            │
│  │  导出/导入   │                                            │
│  │  (可选)     │                                            │
│  └──────┬──────┘                                            │
└─────────┼───────────────────────────────────────────────────┘
          │
          │  webhook / API
          v
┌─────────────────────────────────────────────────────────────┐
│                    GitHub 仓库                              │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────────────┐ │
│  │   代码资产   │  │ contract.yaml│  │   catalog/          │ │
│  │   (src/)    │  │  (导出副本)  │  │   (可选归档)        │ │
│  └─────────────┘  └─────────────┘  └─────────────────────┘ │
└─────────────────────────────────────────────────────────────┘
```

## 四个阶段

### 第一阶段：建立连接

让 CLI 具备感知服务端的能力。

**关键交付**：
- Repository 接口抽象（本地/远程统一接口）
- 登录认证（OAuth / API Key）
- 自动切换逻辑（有配置走远程，无配置走本地）

### 第二阶段：数据迁移

实现混合模式，将服务端设为权威，同时保留 Git 可见性。

**关键交付**：
- 契约拉取/推送（`contract pull` / `contract push`）
- Webhook 闭环（GitHub 推送自动同步到服务端）
- 冲突解决策略（服务端优先/仓库优先/人工合并）

### 第三阶段：能力上移

将重计算任务迁移到服务端，CLI 作为指令发起者。

**关键交付**：
- 远程发现（服务端扫描多源）
- 快照版本化（增量快照 + 历史库）
- 治理策略下发（规则集中管理，CLI 同步）

### 第四阶段：质量门禁

引入契约覆盖率概念，支持 CI 流程中的质量门禁。

**关键交付**：
- 覆盖率检查命令（`check --fail-under=80`）
- GitHub Check Run 集成（PR 自动检查）
- 覆盖率趋势追踪（Dashboard 展示）

## 关键决策

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 契约事实源 | 服务端数据库 | 支持实时协作、权限控制 |
| 仓库契约文件 | 导出副本 | 保留 Git 可见性，但非权威 |
| CLI 默认模式 | 在线模式 | 简化使用，实时反馈 |
| 离线支持 | 缓存机制 | 兼顾网络不稳定场景 |
| 快照存储 | 服务端历史库 | 集中管理，便于查询对比 |
| 冲突解决 | 服务端优先 | 保证权威性的同时提供反馈 |
| 快照归档 | 全量 + 增量 | 平衡存储成本与查询效率 |

## 演化时间线

| 阶段 | 目标 | 关键交付 | 预计周期 |
|------|------|---------|---------|
| 第一阶段 | 建立连接 | Repository 抽象、登录认证 | 1-2 周 |
| 第二阶段 | 数据迁移 | Pull/Push、Webhook 闭环 | 2-3 周 |
| 第三阶段 | 能力上移 | 远程发现、策略下发 | 3-4 周 |
| 第四阶段 | 质量门禁 | `check` 命令、CI 集成 | 2-3 周 |

**总计**：8-12 周完成从 CLI 到平台的演化。
